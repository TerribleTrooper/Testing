ЗАДАНИЕ 3. ПРОЕКТИРОВАНИЕ АРХИТЕКТУРЫ

========================================
1. ОБЩЕЕ ОПИСАНИЕ
========================================

Система предназначена для:
- отображения партнеров
- работы с корзиной
- отправки push-уведомлений пользователю

Архитектура построена по принципу клиент-серверного взаимодействия
с использованием микросервисного подхода.

========================================
2. ОСНОВНЫЕ КОМПОНЕНТЫ
========================================

1. Mobile App (клиент)
2. API Gateway
3. Partner Service
4. Cart Service
5. Notification Service
6. Database
7. Message Broker (очередь сообщений)
8. Push Provider (Firebase / APNS)

========================================
3. АРХИТЕКТУРНАЯ СХЕМА
========================================

[ Mobile App ]
        ↓
[ API Gateway ]
        ↓
 ┌───────────────┬───────────────┐
 ↓               ↓               ↓
[Partner Service][Cart Service][Notification Service]
        ↓               ↓
     [Database]     [Database]

Notification Service:
        ↓
[ Message Broker ]
        ↓
[ Push Provider (Firebase / APNS) ]

========================================
4. ОПИСАНИЕ КОМПОНЕНТОВ
========================================

Mobile App:
- отправляет HTTP-запросы
- отображает данные пользователю

API Gateway:
- единая точка входа
- маршрутизация запросов
- может выполнять авторизацию и логирование

Partner Service:
- хранит и отдает список партнеров
- отвечает за данные о доставке

Cart Service:
- управляет корзиной пользователя
- хранит состав корзины
- применяет бизнес-правила (лимиты, цена)

Notification Service:
- отвечает за отправку уведомлений
- получает события и отправляет push

Database:
- хранение данных (партнеры, корзина)

Message Broker:
- асинхронная передача событий
- уменьшает связанность сервисов

Push Provider:
- внешний сервис (Firebase / Apple Push Notification Service)

========================================
5. СЦЕНАРИЙ РАБОТЫ (ПРИМЕР)
========================================

1. Пользователь открывает приложение
2. Mobile App отправляет запрос:
   GET /api/v1/partners

3. API Gateway направляет запрос в Partner Service

4. Partner Service возвращает список партнеров

5. Пользователь добавляет товар в корзину

6. Запрос идет в Cart Service

7. Cart Service:
   - проверяет лимиты
   - сохраняет данные

8. При необходимости отправки уведомления:
   - создается событие
   - отправляется в Message Broker

9. Notification Service:
   - получает событие
   - отправляет push через Firebase/APNS

========================================
6. ПОЧЕМУ ВЫБРАНО ТАКОЕ РЕШЕНИЕ
========================================

- Разделение на сервисы упрощает масштабирование
- API Gateway централизует обработку запросов
- Message Broker позволяет работать асинхронно
- Notification Service изолирован от бизнес-логики
- Система легко расширяется (новые сервисы)

========================================
7. ДОПОЛНИТЕЛЬНО (ОПЦИОНАЛЬНО)
========================================

- Возможность кеширования (Redis)
- Логирование и мониторинг
- Балансировка нагрузки
- Горизонтальное масштабирование сервисов
